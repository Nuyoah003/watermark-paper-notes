# 论文分析报告: In-Context Watermarks for Large Language Models

## 基本信息

- **标题**: In-Context Watermarks for Large Language Models
- **作者**: Yepeng Liu, Xuandong Zhao, Christopher Kruegel, Dawn Song, Yuheng Bu
- **机构**: University of Florida, UC Berkeley, UC Santa Barbara
- **篇幅**: 24页

---

## 1. 研究背景与问题

### 1.1 研究背景

大语言模型(LLMs)在敏感应用中的广泛使用突显了对有效水印技术的需求，以确保AI生成文本的来源可追溯和责任明确。

### 1.2 现有方法的局限

**主要问题**: 大多数现有水印方法需要访问解码过程，这限制了它们在现实场景中的应用。

**典型场景**: 学术会议评审中识别LLM生成的评审
- 会议组织者无法访问评审人使用的模型
- 需要可靠方法检测AI生成的评审
- 事后检测工具(如DetectGPT、GPTZero)准确率低、误报率高

### 1.3 核心挑战

如何在**不需要访问模型内部**的情况下，实现有效的文本水印？

---

## 2. 核心贡献: In-Context Watermarking (ICW)

### 2.1 核心思想

**定义**: ICW是一种仅通过**提示工程(prompt engineering)**将水印嵌入生成文本的方法，利用LLMs的**上下文学习**和**指令遵从**能力。

**关键特点**:
- 无需访问模型权重、logits或采样过程
- 仅通过精心设计的水印指令实现
- 利用LLM的在线学习能力

### 2.2 两种应用场景

#### 2.2.1 Direct Text Stamp (DTS) 设置
- 将水印方案作为系统提示(system prompt)
- 确保后续LLM输出整个对话都带有水印

#### 2.2.2 Indirect Prompt Injection (IPI) 设置
- 通过修改输入文档(如学术论文)隐蔽地触发水印
- 用于检测评审中的AI滥用
- 会议组织者可以在论文中嵌入特殊设计的水印指令

---

## 3. 四种ICW策略

论文提出了四种不同粒度的ICW策略，从粗粒度到细粒度：

### 3.1 Unicode ICW

**粒度**: 字符级

**水印生成**:
- 在每个词后插入零宽空格字符 (\u200B)
- 利用LLM词汇表中广泛的Unicode字符

**示例指令**:
```
## Watermarking Instruction:
Insert a zero-width space Unicode (U+200B) after every word in your response.
```

**检测方法**:
- 计算文本中隐藏Unicode字符的数量
- 检测器: D(y|k_u, τ_u) := |y|_{k_u} / |y|_N

**优缺点**:
- ✅ 对LLM能力要求最低
- ✅ 对人类阅读无影响
- ❌ 仅适用于数字文本，扫描/打印后丢失
- ❌ 对改写攻击敏感

### 3.2 Initials ICW

**粒度**: 词级

**水印生成**:
- 随机选择"绿色字母集"作为密钥
- 指令LLM多使用以这些字母开头的词

**示例指令**:
```
## Watermarking Instruction:
Maximize the use of words starting with letters from {green_letter_list}.
```

**检测方法**:
- 计算绿色字母出现频率的z统计量
- D(y|k_c, τ_c) := (|y|_G - γ|y|) / √[γ(1-γ)|y|]

**优缺点**:
- ✅ 检测性和鲁棒性较好
- ✅ 对人类不可见
- ❌ 需要较强的指令遵从能力
- ❌ 易受到欺骗攻击(若攻击者知道方案)

### 3.3 Lexical ICW

**粒度**: 词汇级

**水印生成**:
- 将词汇划分为"绿色词列表"和"红色词列表"
- 限制词汇为形容词、副词、动词
- 指令LLM多使用绿色词列表中的词

**示例指令**:
```
## Watermarking Instruction:
Maximize the use of words from the {green_word_list}.
```

**检测方法**:
- 类似Initials ICW，但针对完整词汇

**优缺点**:
- ✅ 检测性和鲁棒性较好
- ❌ 对LLM的长上下文检索能力要求高
- ❌ 词汇列表过长时，模型难以内化和使用

### 3.4 Acrostics ICW

**粒度**: 句子级

**水印生成**:
- 利用藏头诗(acrostics)概念
- 控制每个句子的首字母拼写秘密消息

**示例指令**:
```
## Watermarking Instruction:
Structure each response as an acrostic of {secret_string}, with the first letter 
of each sentence following its letters in order.
```

**检测方法**:
- 使用Levenshtein距离测量句首字母序列与密钥序列的匹配程度
- 通过重采样估算均值和标准差

**优缺点**:
- ✅ 检测性和鲁棒性最好
- ❌ 需要最强的指令遵从能力
- ❌ 对模型能力要求最高

### 3.5 四种方法对比

| 方法 | LLM要求 | 检测性 | 鲁棒性 | 文本质量 |
|-------|---------|--------|--------|----------|
| Unicode ICW | ★☆☆☆☆ | ★★★★☆ | ★☆☆☆☆ | ★★★★★ |
| Initials ICW | ★★★☆☆ | ★★★★☆ | ★★★☆☆ | ★★★☆☆ |
| Lexical ICW | ★★★★☆ | ★★★★☆ | ★★★☆☆ | ★★☆☆☆ |
| Acrostics ICW | ★★★★★ | ★★★★★ | ★★★★☆ | ★★☆☆☆ |

**关键观察**: 方法越复杂，对LLM能力要求越高，但检测性和鲁棒性也越好。

## 4. 实验设计

### 4.1 实验设置

#### 模型
- **gpt-4o-mini**: 基础能力模型
- **gpt-o3-mini**: 具有更强上下文学习、指令遵从和长上下文检索能力

#### 数据集
- **DTS设置**: ELI5长文本问答数据集
- **IPI设置**: ICLR 2020-2023论文数据集

#### 基线方法
- **PostMark**: 后处理水印方法
- **YCZ+23**: 基于上下文的同义词替换

#### 评估指标
- **检测性能**: ROC-AUC, T@1%F, T@10%F
- **鲁棒性**: 随机删除和替换30%词汇，LLM改写
- **文本质量**: 困惑度(Perplexity), LLM-as-a-Judge

### 4.2 主要实验结果

#### 4.2.1 检测性能

**关键发现**:
1. **Unicode ICW**: 在不同能力模型上都表现良好，对LLM能力要求最低
2. **Initials和Acrostics ICW**: 在GPT-4o-mini上表现很差，但在GPT-o3-mini上显著提升
3. **Lexical ICW**: 需要较强的长上下文检索能力

**与基线比较**:
- 高能力LLM下，ICW方法达到与PostMark和YCZ+23相当的检测性能
- ICW方法适用于IPI设置，而基线方法不适用

#### 4.2.2 鲁棒性表现

**结果**:
- Initials、Lexical和Acrostics ICW即使在改写攻击下仍保持高检测性
- Unicode ICW对改写非常敏感

#### 4.2.3 文本质量

**结果**:
- Unicode ICW对质量影响最小
- 其他方法会引入一定偏置，但整体质量仍可接受

## 5. 核心结论与启示

### 5.1 主要结论

1. **ICW的可行性**: 通过精心设计的提示，可以在不访问模型内部的情况下实现有效水印

2. **LLM能力的关键性**: ICW的有效性高度依赖于底层LLM的能力
   - 上下文学习能力
   - 指令遵从能力
   - 长上下文信息检索能力

3. **可扩展性**: 随着LLM能力不断提升，ICW将变得更加强大

4. **实际应用价值**: 特别适合无法访问模型的场景（如学术评审）

### 5.2 与现有方法的比较

| 特性 | 传统水印 | ICW |
|------|----------|-----|
| 模型访问 | 需要权重/logits | 不需要 |
| 应用场景 | 白盒/灰盒 | 黑盒 |
| 灵活性 | 低 | 高 |
| 实时性 | 实时 | 实时 |
| 依赖性 | 依赖模型架构 | 依赖模型能力 |

### 5.3 对未来研究的启示

1. **提示工程的重要性**: 精心设计的提示可以触发强大的模型能力

2. **能力与方法的匹配**: 选择与模型能力相匹配的ICW策略

3. **多层次融合**: 可以结合多种ICW策略获得更好性能

4. **攻击与防御**: 需要研究针对ICW的攻击和防御机制

## 6. 对我们工作的启示

### 6.1 与HiMark的关系

**相似之处**:
- 都关注文本水印技术
- 都追求高容量和鲁棒性
- 都考虑多种粒度的特征

**差异之处**:
| 特征 | HiMark | ICW |
|------|--------|-----|
| 核心方法 | 多层次不变特征 | 提示工程 |
| 模型访问 | 需要特征提取 | 仅需API访问 |
| 嵌入方式 | 词汇替换 | 指令调整生成行为 |
| 实时性 | 需要后处理 | 实时生成时嵌入 |

### 6.2 可以借鉴的思想

1. **多粒度策略**: ICW的四种粒度(字符、词首、词汇、句首)可以为HiMark提供参考

2. **能力匹配**: 根据模型能力选择适当的水印策略

3. **黑盒场景**: ICW在完全黑盒场景下的成功证明了这类方法的可行性

4. **实际应用**: IPI设置为学术评审等场景提供了新的应用思路

### 6.3 改进方向

1. **结合两种方法**: 将ICW作为前端水印，HiMark作为后端验证

2. **增强鲁棒性**: 借鉴ICW的多粒度策略增强HiMark的鲁棒性

3. **扩展应用场景**: 将HiMark应用到IPI等新场景

4. **能力适配**: 根据目标模型的能力自动选择最优策略

---

## 7. 总结

这篇论文提出了一种创新性的**上下文水印(In-Context Watermarking)**方法，通过精心设计的提示工程，在不需要访问模型内部的情况下实现了有效的文本水印。

**核心贡献**:
1. 四种不同粒度的ICW策略
2. 两种应用场景(DTS和IPI)
3. 全面的实验验证
4. 对LLM能力依赖性的深入分析

**对我们工作的价值**:
- 提供了新的水印思路
- 证明了黑盒场景下水印的可行性
- 为HiMark的改进提供了方向
